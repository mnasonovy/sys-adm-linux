# Лабораторная работа № 4. DevOps‑автоматизация: systemd, bash‑скрипты и Ansible

**Среда:** VirtualBox или VMware. **Образ:** Ubuntu Server. Можно использовать ВМ и пользователя из ЛР № 1–3.

**Требование:** для всех этапов автоматизации (bash‑скрипты, systemd, Ansible) необходимы скриншоты или выводы ключевых команд. Скрипты, unit‑файлы и материалы Ansible нужно **версионировать в Git** и **опубликовать в публичном** удалённом репозитории (GitHub, GitLab, Codeberg и т.п.); в отчёт — ссылка на репозиторий. Секреты (приватные SSH‑ключи, пароли) в репозиторий не помещать.

---

## Задания

1. **Подготовка ВМ и пользователя**
   - Развернуть (или использовать) ВМ Ubuntu Server.
   - Обеспечить доступ по SSH по ключу под пользователем с правами `sudo`.
   - Установить `git`, `curl`, `python3`, `python3-venv`, `ansible`.

2. **Bash‑скрипт сервиса**
   - Создать каталог `/opt/lab4-service/html`.
   - Написать `service.sh` (с `#!/usr/bin/env bash`, `set -euo pipefail`), запускающий HTTP‑сервер (например, `python3 -m http.server`) и отдающий страницу с фамилией студента.
   - Проверить доступность страницы по `http://<IP_ВМ>:8000/`.

3. **systemd‑юнит**
   - Создать unit‑файл `lab4-service.service` в `/etc/systemd/system`.
   - Настроить автозапуск, перезапуск при сбоях, запуск от пользователя `www-data`.
   - Проверить `systemctl enable --now lab4-service`, `systemctl status lab4-service`.

4. **Логирование и healthcheck**
   - Просмотреть логи через `journalctl -u lab4-service`.
   - Написать скрипт `/usr/local/bin/lab4-healthcheck.sh`, выполняющий HTTP‑проверку сервиса (код 200).
   - Добиться корректного результата проверки при работающем/остановленном сервисе.

5. **Git‑репозиторий конфигурации**
   - Создать каталог проекта (например, `~/lab4-devops`).
   - Скопировать туда `service.sh`, пример unit‑файла и healthcheck‑скрипта; добавить краткий `README.md`.
   - Инициализировать репозиторий `git`, коммитить изменения по мере работы; к концу лабораторной в репозитории должны быть все артефакты, включая Ansible из заданий 6–8.
   - Не добавлять в коммиты приватные ключи и пароли: для сдачи хранить в репозитории **пример** инвентаря (`inventory.example.ini` и т.п.) или описание переменных; при необходимости исключить локальный `inventory.ini` через `.gitignore`.
   - Создать **публичный** репозиторий на выбранном хостинге, добавить `remote`, выполнить `git push`; убедиться, что `git status` чистый.

6. **Ansible‑инвентарь**
   - Создать рабочий `inventory.ini` (локально) с группой `lab4` и хостом, указывая IP ВМ и пользователя.
   - Настроить использование SSH‑ключа.
   - Проверить модулем `ansible -i inventory.ini lab4 -m ping`.
   - Закоммитить в Git безопасный вариант инвентаря для проверяющего (шаблон без секретов), как указано в задании 5.

7. **Ansible‑playbook развёртывания сервиса**
   - Написать `site.yml`, который:
     - устанавливает необходимые пакеты;
     - создаёт каталоги сервиса;
     - разворачивает `index.html` с фамилией студента;
     - копирует `service.sh`;
     - создаёт unit‑файл `lab4-service.service`;
     - выполняет `daemon-reload` и запускает/включает сервис.
   - Убедиться, что повторный запуск playbook’а идемпотентен (нет неожиданных изменений).

8. **Healthcheck через Ansible**
   - Добавить в `site.yml` задачу `uri`, которая проверяет доступность сервиса (HTTP 200) по `http://127.0.0.1:8000/`.
   - Снова выполнить playbook и зафиксировать результат проверки.
   - Закоммитить актуальный `site.yml` и при необходимости обновить публичный репозиторий (`git push`).

## Шаблон отчёта

1. **Среда выполнения**
   - Гипервизор (VirtualBox / VMware), версия.
   - Образ: Ubuntu Server (версия).
   - IP‑адрес ВМ, используемый пользователь.

2. **Задание 1. Подготовка ВМ**
   - Краткое описание шага; выводы команд установки пакетов.

3. **Задание 2. Bash‑скрипт**
   - Содержимое `service.sh`.
   - Скриншот или вывод `curl http://<IP_ВМ>:8000/`.

4. **Задание 3. systemd**
   - Содержимое `lab4-service.service`.
   - Вывод `systemctl status lab4-service`.

5. **Задание 4. Логи и healthcheck**
   - Вывод `journalctl -u lab4-service -n 20`.
   - Содержимое `lab4-healthcheck.sh` и пример запуска (OK/FAIL).

6. **Задание 5. Git и публичный репозиторий**
   - Ссылка на **публичный** репозиторий (HTTPS или веб‑страница проекта).
   - Скриншот или вывод `git status`, `git log --oneline`.

7. **Задание 6–8. Ansible**
   - Содержимое `inventory.ini` (без приватных ключей).
   - Содержимое `site.yml` (основные части).
   - Вывод `ansible -i inventory.ini lab4 -m ping`.
   - Вывод `ansible-playbook -i inventory.ini site.yml` (как минимум один успешный прогон).

8. **Выводы**
   - Кратко: что автоматизировано, какие DevOps‑практики использовались, с какими трудностями столкнулись.

## Критерии оценивания

| Задание | Критерий |
|--------|----------|
| 1 | ВМ подготовлена, SSH по ключу работает, необходимые пакеты установлены |
| 2 | Bash‑скрипт сервиса корректно запускает HTTP‑сервер и отдаёт страницу с фамилией студента |
| 3 | systemd‑юнит `lab4-service` создан, сервис стартует автоматически и управляется через `systemctl` |
| 4 | Логи сервиса просматриваются через `journalctl`, healthcheck‑скрипт корректно определяет работоспособность сервиса |
| 5 | Скрипты, unit‑файлы и материалы Ansible версионированы в Git, репозиторий **публично** опубликован; в репозитории нет секретов; `git status` чистый |
| 6 | Подготовлен Ansible‑инвентарь, модуль `ping` успешно отрабатывает |
| 7 | Ansible‑playbook устанавливает пакеты, разворачивает файлы и запускает сервис; повторный запуск идемпотентен |
| 8 | Healthcheck через Ansible (модуль `uri`) успешно выполняется (HTTP 200) |

Лабораторная работа засчитывается при выполнении всех заданий и наличии отчёта с командами, конфигурационными файлами и выводами.
