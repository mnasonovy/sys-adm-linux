## Задание 3. Настройка сетевого интерфейса

На данном этапе была выполнена настройка сетевого интерфейса виртуальной машины с использованием утилиты **netplan**.

В качестве сетевого подключения использовался режим **сетевого моста (Bridge Adapter)**, что позволило виртуальной машине получить IP-адрес в одной сети с хостовой системой.

### Проверка сетевых интерфейсов

Для просмотра доступных сетевых интерфейсов была выполнена команда:

```
ip a
```

В результате был обнаружен интерфейс `enp0s3`.
![img_7.png](images/lab2/img_7.png)

### Настройка netplan

Для конфигурации сети был отредактирован файл:

```
sudo nano /etc/netplan/50-cloud-init.yaml
```

В файл была добавлена следующая конфигурация:

```yaml
network:
  version: 2
  ethernets:
    enp0s3:
      dhcp4: no
      addresses:
        - 192.168.0.150/24
      routes:
        - to: default
          via: 192.168.0.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 1.1.1.1
```

![img_8.png](images/lab2/img_8.png)
После внесения изменений конфигурация была применена:

```
sudo netplan apply
```

### Отключение автоматической настройки сети (cloud-init)

В процессе настройки было выявлено, что после перезагрузки виртуальной машины конфигурация сети сбрасывается.
Причиной этого является утилита **cloud-init**, которая автоматически генерирует сетевые настройки.

Для предотвращения перезаписи конфигурации было выполнено отключение сетевой настройки cloud-init.

Для этого был создан и отредактирован файл:

```
sudo nano /etc/cloud/cloud.cfg.d/99-disable-network-config.cfg
```

В файл была добавлена следующая конфигурация:

```yaml
network: {config: disabled}
```

После этого изменения конфигурации сети через netplan сохраняются после перезагрузки системы.

## Задание 5. Создание RAID-массива уровня 5

На данном этапе свободные неразмеченные диски виртуальной машины были объединены в RAID-массив 5-го уровня.

### Установка утилиты mdadm

Для работы с RAID была установлена утилита:

```id="m8k2fa"
sudo apt update
sudo apt install mdadm -y
```

### Создание RAID-массива

RAID-массив был создан из трёх дисков (`sdb`, `sdc`, `sdd`) с использованием следующей команды:

```id="x2v9rt"
sudo mdadm --create --verbose /dev/md0 \
--level=5 \
--raid-devices=3 \
/dev/sdb /dev/sdc /dev/sdd
```
![img_14.png](images/lab2/img_14.png)

### Проверка состояния массива

Для проверки состояния RAID-массива была использована команда:

```

```

В результате было подтверждено, что массив `md0` успешно создан и находится в активном состоянии.
![img_15.png](images/lab2/img_15.png)

### Детальная информация о массиве

Дополнительно была получена подробная информация о RAID-массиве:

```

```

Команда отобразила параметры массива, включая уровень RAID, количество устройств и состояние дисков.

## Задание 6. Настройка LVM и монтирование

На данном этапе на основе RAID-массива был настроен менеджер логических томов (LVM).

Для создания физического тома была выполнена команда:

### Создание группы томов (VG)

Группа томов была создана с именем по фамилии студента:

```
sudo vgcreate nasonov /dev/md0
```

Проверка:

```
sudo vgs
```

![img_18.png](images/lab2/img_18.png)

### Создание логического тома (LV)

Был создан логический том `lv0`, занимающий всё доступное пространство:

```
sudo lvcreate -n lv0 -l 100%FREE nasonov
```

Проверка:

```
sudo lvs
```
![img_19.png](images/lab2/img_19.png)

### Создание файловой системы

На логическом томе была создана файловая система:


### Монтирование

Была создана точка монтирования:

```
sudo mkdir -p /raid/0
```

Выполнено монтирование:

```
sudo mount /dev/nasonov/lv0 /raid/0
```

Проверка:

```

```
![img_21.png](images/lab2/img_21.png)

![img_22.png](images/lab2/img_22.png)
После этого была выполнена проверка:

Ошибок при выполнении команды не возникло, что подтверждает корректность настроек.

```bash

```

Применение настроек:

```

```
![img_67.png](images/lab2/img_67.png)

### Настройка проброса портов

Были настроены правила DNAT для перенаправления трафика с портов 139 и 445 на VPN-клиент:

```
sudo iptables -t nat -A PREROUTING -i enp0s3 -p tcp --dport 139 -j DNAT --to-destination 10.8.0.2:139
sudo iptables -t nat -A PREROUTING -i enp0s3 -p tcp --dport 445 -j DNAT --to-destination 10.8.0.2:445
```

### Разрешение форвардинга

```
sudo iptables -A FORWARD -p tcp -d 10.8.0.2 --dport 139 -j ACCEPT
sudo iptables -A FORWARD -p tcp -d 10.8.0.2 --dport 445 -j ACCEPT
```

### Настройка NAT

```
sudo iptables -t nat -A POSTROUTING -o tun0 -j MASQUERADE
```

# Выводы

В ходе выполнения лабораторной работы были получены практические навыки администрирования Linux-систем и настройки сетевой инфраструктуры.

Были выполнены следующие задачи:

* развернута и настроена виртуальная машина;
* настроен SSH-доступ по ключу с отключением парольной аутентификации;
* сконфигурирована сеть с использованием netplan и статического IP-адреса;
* создан RAID-массив уровня 5 и настроен менеджер логических томов LVM;
* организовано постоянное монтирование файловой системы;
* настроен доступ по NFS и реализовано резервное копирование с использованием cron;
* выполнена настройка межсетевого экрана (iptables, ufw);
* реализован мониторинг сети и анализ трафика;
* настроен VPN-сервер OpenVPN и проверено подключение клиента;
* настроен Samba-сервер и обеспечен доступ к общим ресурсам;
* выполнена проверка доступности сервисов и портов из внешней сети.

В процессе выполнения работы возникли следующие трудности:

* проблемы с доступом по SSH после изменения настроек аутентификации и блокировки порта, что потребовало восстановления доступа через локальную консоль;
* некорректная работа `ssh-copy-id` в среде Windows, из-за чего ключ пришлось добавлять вручную;
* сброс настроек сети после перезагрузки из-за работы cloud-init;
* сложности с настройкой RAID-массива (переименование устройства md0 → md127 после перезагрузки);
* ошибки при настройке OpenVPN, связанные с несовпадением сертификата и ключа сервера;
* особенности сетевого взаимодействия между WSL, хостом и виртуальной машиной, из-за чего возникали проблемы при проверке доступности портов;
* трудности с пониманием маршрутизации и проброса портов между различными сетевыми интерфейсами;
* временные ошибки при настройке брандмауэра, приводившие к недоступности сервисов.

Все возникшие проблемы были проанализированы и устранены в процессе выполнения работы.

В результате лабораторной работы была развернута полноценная тестовая инфраструктура, включающая файловый сервер, VPN, средства резервного копирования и механизмы сетевой безопасности.
