# Среда выполнения

В качестве среды выполнения использовалась система **WSL2**, внутри которой был развёрнут Linux-хост на базе **Ubuntu 22.04**, выступающий в роли KVM-хоста.

Важно отметить, что в рамках данной работы под хостовой системой понимается именно **Linux-окружение внутри WSL2**, в котором осуществляется работа с гипервизором **KVM/libvirt**, а не базовая операционная система Windows.

Использование WSL2 позволило обеспечить доступ к аппаратной виртуализации (через `/dev/kvm`) и организовать работу гипервизора без развёртывания отдельной физической или полноценной виртуальной машины.

## Задание 2. Создание гостевой ВМ в KVM

Для создания гостевой виртуальной машины в условиях консольной среды **WSL2** был использован headless‑подход с применением утилиты `virt-install` и образа **Ubuntu Server cloud image**.

Данный способ позволяет полностью автоматизировать процесс развёртывания ВМ без использования графического интерфейса и является стандартным подходом при работе с KVM в серверных или минимальных окружениях.

### Подготовка каталога виртуальной машины

Перед созданием ВМ был подготовлен отдельный каталог для хранения её образов и вспомогательных файлов:

```
VM_NAME=ubuntu-lab3
IMG_DIR=/var/lib/libvirt/images/$VM_NAME
BASE_IMG=$IMG_DIR/ubuntu-server.qcow2
SEED_IMG=$IMG_DIR/seed.iso

sudo mkdir -p "$IMG_DIR"
```

Такое разделение позволяет изолировать файлы конкретной виртуальной машины и упростить дальнейшее администрирование.

### Загрузка и подготовка образа Ubuntu Server

В качестве базового образа использовался официальный cloud image Ubuntu Server 22.04 (Jammy):

```
cd /tmp
wget -O jammy-server-cloudimg-amd64.img   https://cloud-images.ubuntu.com/jammy/current/jammy-server-cloudimg-amd64.img
sudo cp jammy-server-cloudimg-amd64.img "$BASE_IMG"
sudo qemu-img resize "$BASE_IMG" 20G
```

После копирования образ был расширен до 20 ГБ с помощью `qemu-img resize`, что позволило увеличить доступное дисковое пространство внутри гостевой системы.

![img_3.png](images/lab3/img_3.png)

### Подготовка cloud-init для первичной настройки

Для автоматической начальной конфигурации ВМ использовался механизм **cloud-init**, позволяющий задать параметры системы при первом запуске.

Были созданы файлы `user-data` и `meta-data`, содержащие:

- имя хоста;
- создание пользователя `user`;
- добавление пользователя в группу `sudo`;
- настройку пароля;
- установку базовых пакетов (включая `qemu-guest-agent`).

```
cat > user-data <<'EOL'
#cloud-config
hostname: ubuntu-lab3
manage_etc_hosts: true
users:
  - default
  - name: user
    groups: sudo
    shell: /bin/bash
    sudo: ALL=(ALL) NOPASSWD:ALL
    lock_passwd: false
chpasswd:
  list: |
    user:User12345!
  expire: false
ssh_pwauth: true
package_update: true
packages:
  - qemu-guest-agent
EOL

cat > meta-data <<'EOL'
instance-id: ubuntu-lab3
local-hostname: ubuntu-lab3
EOL

cloud-localds seed.iso user-data meta-data
sudo mv seed.iso "$SEED_IMG"
rm -f user-data meta-data
```

Использование cloud-init позволяет избежать ручной настройки системы после установки и значительно ускоряет процесс развёртывания виртуальной машины.

![img_4.png](images/lab3/img_4.png)

### Создание и импорт виртуальной машины

Виртуальная машина была создана с использованием `virt-install` в режиме импорта готового образа:

```
sudo virt-install   --name "$VM_NAME"   --memory 2048   --vcpus 2   --import   --os-variant ubuntu22.04   --disk path="$BASE_IMG",format=qcow2,bus=virtio   --disk path="$SEED_IMG",device=cdrom   --network network=default,model=virtio   --graphics none   --console pty,target_type=serial   --noautoconsole
```

Ключевые параметры:
- `--import` — использование уже подготовленного образа без установки через ISO;
- `virtio` — ускоренный драйвер для дисков и сети;
- `--graphics none` — работа без графического интерфейса;
- `--console` — подключение через последовательную консоль.

## Задание 6. Проброс дополнительного диска в гостевую ВМ через KVM

На данном этапе был создан и подключён дополнительный виртуальный диск к гостевой ВМ.  
Это позволяет расширить дисковую подсистему и отработать сценарий управления дополнительными устройствами хранения в среде виртуализации.

### Создание виртуального диска на хосте

Перед созданием диска была задана переменная с именем виртуальной машины:

```
VM_NAME=ubuntu-lab3
```

Для хранения образов виртуальных дисков используется стандартный каталог libvirt:

```
sudo mkdir -p /var/lib/libvirt/images
```

Далее был задан путь к создаваемому диску:

```
DATA_IMG=/var/lib/libvirt/images/${VM_NAME}-data.qcow2
echo $DATA_IMG
```

После этого был создан диск формата **qcow2** объёмом 10 ГБ:

```
sudo qemu-img create -f qcow2 "$DATA_IMG" 10G
```

Формат **qcow2** поддерживает динамическое выделение места (thin provisioning), благодаря чему файл изначально занимает минимальный объём и увеличивается по мере записи данных.

![img_16.png](images/lab3/img_16.png)

### Подключение диска к гостевой ВМ

Созданный диск был подключён к виртуальной машине с помощью команды:

```
sudo virsh attach-disk "$VM_NAME" "$DATA_IMG" vdb --persistent
```

Параметры команды:

- `vdb` — имя устройства внутри гостевой системы;
- `virtio` (используется по умолчанию) — паравиртуализированный драйвер, обеспечивающий более высокую производительность;
- `--persistent` — делает подключение постоянным, сохраняя его в конфигурации ВМ.

В результате был обнаружен новый диск `vdb`, что подтверждает успешное подключение устройства.

## Задание 7. Создание GPT-разметки, раздела и файловой системы ext4

### Разметка диска

На гостевой ВМ:

```

```

Использование GPT обеспечивает поддержку современных систем и корректную работу с дисками различного объёма.

## Задание 8. Обеспечение доступности `/disk` пользователю `user`

На данном этапе была выполнена настройка прав доступа к каталогу `/disk`, в который смонтирован дополнительный раздел.  
Цель — предоставить пользователю `user` возможность полноценно работать с файловой системой (создавать, изменять и удалять файлы).

### Настройка прав доступа

По умолчанию после монтирования владельцем каталога является пользователь `root`, что ограничивает доступ для обычных пользователей.  
Для передачи прав управления каталогом пользователю `user` были выполнены следующие команды:

```
sudo chown user:user /disk
sudo chmod 755 /disk
```

Пояснение:

- `chown user:user` — устанавливает пользователя `user` владельцем каталога;
- `chmod 755` — задаёт права доступа:
  - владелец — полный доступ (чтение, запись, выполнение);
  - остальные — только чтение и выполнение.

Таким образом, пользователь `user` получает полный контроль над каталогом `/disk`.

### Проверка под пользователем `user`

Для проверки корректности настроек был выполнен вход под пользователем `user` и проведены тестовые операции:

```
su - user
touch /disk/test_file
ls -l /disk
rm /disk/test_file
```

В результате:

- файл успешно создаётся в каталоге `/disk`;
- файл корректно отображается в списке;
- файл без ошибок удаляется.

Это подтверждает, что пользователь `user` имеет необходимые права для работы с файловой системой.

На данном этапе в гостевой системе была выполнена установка и базовая настройка **Docker**.


В результате:

В результате:

- отображается версия установленного Docker;
- контейнер `hello-world` успешно скачивается и запускается;
- вывод сообщения `Hello from Docker!` подтверждает корректную работу Docker и доступ пользователя к управлению контейнерами.

Таким образом, Docker установлен и настроен, а пользователь `user` может запускать контейнеры без повышения привилегий.

## Задание 10. Развёртывание контейнера nginx, выводящего фамилию студента

На данном этапе был реализован сценарий публикации статического веб-контента с использованием контейнера **nginx**.  
В качестве корня сайта использовался каталог, расположенный на ранее смонтированном разделе `/disk`, что демонстрирует интеграцию контейнеров с файловой системой гостевой ВМ.

### Подготовка каталога сайта

На гостевой ВМ был создан каталог для размещения веб-страницы:

```
mkdir -p /disk/site
```

Размещение данных именно в `/disk` позволяет использовать дополнительный диск, подключённый ранее, что соответствует требованиям лабораторной работы.

### Создание HTML-страницы

В каталоге был создан файл `index.html` со статическим содержимым:

```
cat > /disk/site/index.html <<'EOL'
<!DOCTYPE html>
<html lang="ru">
<head>
    <meta charset="UTF-8">
    <title>Лабораторная работа №3</title>
</head>
<body>
    <h1>Насонов</h1>
    <p>Лабораторная работа №3: KVM и Docker</p>
</body>
</html>
EOL
```

Наличие фамилии студента в HTML-файле является обязательным условием задания.

### Запуск контейнера nginx

Для публикации сайта был запущен контейнер nginx:

```
docker run -d \
  --name lab3-nginx \
  --restart unless-stopped \
  -p 80:80 \
  -v /disk/site:/usr/share/nginx/html:ro \
  nginx
```

Ключевые параметры:

- `-p 80:80` — проброс порта контейнера на порт хоста;
- `-v /disk/site:/usr/share/nginx/html:ro` — монтирование каталога сайта внутрь контейнера;
- `:ro` — режим только для чтения, повышающий безопасность;
- `--restart unless-stopped` — автоматический перезапуск контейнера.

Таким образом, nginx использует содержимое каталога `/disk/site` как корень веб-сервера.

### Проверка работы контейнера

Сначала была выполнена проверка внутри гостевой ВМ:

```
docker ps
curl http://127.0.0.1/
```

Успешный вывод HTML-страницы подтверждает, что контейнер работает корректно и обслуживает HTTP-запросы.

### Проверка доступности с хоста

Так как в данной работе KVM-хостом выступает **Ubuntu в WSL2**, проверка сетевой доступности выполняется с этого окружения:

```
curl http://192.168.122.192/
```

IP-адрес принадлежит внутренней сети `libvirt` (NAT), используемой для взаимодействия с виртуальной машиной.

В ответе сервера корректно отображается HTML-страница, содержащая фамилию студента:

```text
Насонов
```

Это подтверждает:

- корректную работу контейнера nginx;
- успешное монтирование каталога `/disk` внутрь контейнера;
- доступность веб-сервиса по сети.

# Выводы

В ходе выполнения лабораторной работы был подготовлен Linux-хост в среде **WSL2** с поддержкой **KVM/libvirt**, создана и запущена гостевая виртуальная машина **Ubuntu Server**, настроен пользователь `user` и организован защищённый SSH-доступ по ключу.

Также была изучена конфигурация гостевой ВМ, создан и подключён дополнительный виртуальный диск, выполнена разметка в формате **GPT**, создана файловая система **ext4**, настроено постоянное монтирование раздела в каталог `/disk` через `/etc/fstab`, а также выданы права пользователю `user` для работы с этим каталогом.

На завершающем этапе в гостевой системе был установлен **Docker**, выполнена проверка запуска контейнеров и развернут контейнер **nginx**, использующий каталог на смонтированном разделе `/disk` в качестве корня сайта. Проверка с хоста подтвердила доступность веб-страницы и корректное отображение фамилии студента.

Наиболее трудоёмким этапом стал **выбор среды выполнения лабораторной работы**. Попытка использования **VirtualBox** привела к проблемам с поддержкой вложенной виртуализации, из-за чего запуск KVM внутри ВМ оказался нестабильным. В результате было принято решение перейти на **WSL2**, что позволило получить доступ к `/dev/kvm` и корректно использовать аппаратную виртуализацию.

Существенные сложности возникли при работе с дополнительным диском. В процессе настройки было выявлено, что диск подключался в конфигурации ВМ с типом `raw`, несмотря на создание в формате **qcow2**, что приводило к некорректному определению его размера (порядка килобайт вместо 10 ГБ) и ошибкам при разметке. Дополнительно возникали проблемы с кэшированием конфигурации `libvirt` и повторным подключением устройства (`vdb`), что требовало очистки XML-конфигурации ВМ и повторного подключения диска.

Также определённые затруднения были связаны с особенностями сетевой модели **libvirt (NAT)** в связке с WSL2, из-за чего доступ к веб-серверу из Windows-браузера был ограничен и проверка выполнялась с Linux-хоста.

В результате выполнения лабораторной работы были получены практические навыки работы с виртуализацией **KVM**, настройкой и диагностикой среды **WSL2**, управлением дисками и файловыми системами, настройкой безопасного SSH-доступа, а также развёртыванием и использованием контейнеров **Docker**.
